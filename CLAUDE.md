# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev          # Start Vite dev server
npm run build        # Type-check + build for production (also regenerates sitemap/robots.txt)
npm run build-only   # Build without type-check
npm run type-check   # Run vue-tsc type checking only
npm run lint         # ESLint with auto-fix

# Data management (run manually when satellite/transponder data needs refreshing)
npm run fetch-satellites            # Fetch TLE data from Celestrak → public/amateur.txt
npm run fetch-transponders          # Fetch transmitter data from SatNOGS API → public/transponders/
npm run generate-transponder-lookup # Rebuild FM satellite lookup (public/fm-satellites.json)
npm run generate-seo                # Regenerate sitemap.xml and robots.txt
```

There are no tests in this project.

## Architecture

Single-page Vue 3 + TypeScript app. The entire UI is one route (`HomeView.vue` → `MapComponent.vue`). No Pinia/Vuex — state lives in `MapComponent.vue` as `ref`s and is passed down via props/emits.

### Map rendering

OpenLayers (`ol`) renders the map with four stacked layers (lowest to highest z-index):

1. **OSM tile layer** — base map
2. **horizonLayer** (`horizonSource`) — observer's radio horizon circle
3. **lineLayer** (`lineSource`) — line-of-sight arc and orbital path segments
4. **vectorLayer** (`vectorSource`) — satellite icons, horizon footprint polygons, home marker

All map initialization is in `src/components/map/utils/mapSetup.ts`. The `MapLayers` interface (three `VectorSource`s) is passed to every feature class.

### Feature classes (non-Vue)

Plain TypeScript classes that own OpenLayers `Feature` objects and manage their lifecycle:

- **`SatelliteFeature`** — tracks one selected satellite. Manages the satellite icon, its horizon footprint polygon, the curved line-of-sight arc, and the optional 100-minute future path. Updates position every 1 s and path every 10 s via `setInterval`.
- **`NearestSatellitesFeature`** — shows up to 5 nearby satellites when none is selected. Clickable icons that fire an `onSatelliteClick` callback back into `MapComponent.vue`.
- **`HomeLocationFeature`** — home location pin (draggable via OL `Translate` interaction) and the observer's radio horizon `Circle`.

### Page layout

`App.vue` renders `HeaderBar` (fixed 32px height) followed by `RouterView`. Inside `MapComponent.vue`, `.container` is `height: calc(100vh - 32px)` and uses flex-column: the map (`flex: 1`) fills the space above `StatusBar` at the bottom. `body` has `overflow: hidden`.

The version string in `HeaderBar.vue` is a hardcoded date (e.g. `v2026-06-15`) — update it manually on each release.

### UI controls (Vue components)

Four overlay panels are mounted inside `MapComponent.vue`. They are pure UI — they emit events up to `MapComponent.vue` rather than touching the map directly:

- **`HomeLocationControl`** — always visible; set/clear home location (geolocation, map click, manual AGL height).
- **`SatelliteSelector`** — always visible; searchable dropdown to pick a satellite; also shows Baofeng/FM filter toggle.
- **`TransmitterInfoControl`** — mounted only when `selectedSatellite && selectedSatelliteCatalogNumber`; shows transmitter data fetched from `/public/transponders/<NORAD_ID>.json`.
- **`UpcomingSatellitesControl`** — mounted only when `!selectedSatellite && homeCoordinates && upcomingVisibleSatellites.length > 0`; shows pass predictions; clicking an entry selects that satellite.

### Coordinate conventions

- TLE propagation uses `tle.js` (`getLatLngObj`, `getSatelliteInfo`) returning `{ lat, lng, height }`.
- OpenLayers uses EPSG:3857; always convert with `fromLonLat([lon, lat])` before adding features.
- Observer coordinates are stored as `{ lat, lon }` (note: `lon` not `lng`).

### Satellite data flow

1. TLE data: fetched from Celestrak (`gp.php?GROUP=amateur&FORMAT=tle`), cached in `localStorage` for 24 hours (`satgazer_amateur_txt`), falling back to `/public/amateur.txt`. The `fetch-satellites` script regenerates that fallback file.
2. Transmitter data: served from pre-fetched `/public/transponders/<NORAD_ID>.json` files (built by `fetch-transponders.js`); falls back to the SatNOGS API at runtime.
3. FM satellite lookup: `/public/fm-satellites.json` — a `Record<catalogNumber, boolean>` mapping. When Baofeng/FM mode is active (`baofengMode` ref), `loadSatellites()` filters the list to only satellites present in this map. Generated by `generate-transponder-lookup.js`.

### Pass prediction

`predictUpcomingVisibleSatellites()` in `MapComponent.vue` iterates over all satellites, simulating positions at 30-second intervals over a 24-hour window, and collects passes where elevation angle > 0°. It runs whenever `homeCoordinates` or the selected satellite changes and its results populate `UpcomingSatellitesControl`.

### Satellite utility (`src/components/map/utils/satellite.ts`)

Re-exports `getLatLngObj` and `getSatelliteInfo` from `tle.js`, plus `extractCatalogNumber(tle1: string)` which parses the NORAD catalog number from TLE line 1 via regex. All feature classes use this instead of importing `tle.js` directly.

### Math utilities (`src/components/map/utils/calculations.ts`)

Five standalone functions used by feature classes — do not reimplement them:

- `isSatelliteVisible` — line-of-sight check from observer to satellite
- `calculateSatelliteInfo` — returns `{ distance, elevationAngle, azimuth }`
- `calculateSatelliteHorizonPoints` — footprint polygon points for a satellite at altitude
- `createCurvedLine` — great-circle quadratic Bézier for the line-of-sight arc
- `calculateHorizonDistance` — radio horizon distance from observer height (km)

### Persistence

All user settings (home location, selected satellite, show-path toggle, Baofeng mode, cached elevation) are stored in `localStorage` via `src/components/map/utils/settings.ts` (`saveSetting` / `loadSetting` / `removeSetting`). The prefix is `satgazer_`.

### URL parameter

`?id=<NORAD_catalog_number>` — selects a satellite on load and is updated via `history.replaceState` as the user changes selections.

## UX / Style rules

Design tokens are defined as CSS custom properties on `:root` in `src/assets/base.css`. Always use these tokens instead of raw values.

**Blue palette** — use `var(--color-primary)` (`#003C88`) for all interactive buttons. `var(--color-primary-hover)` (`#002D66`) for hover. `var(--color-primary-dim)` for secondary/toggle buttons. Never use `#0078d4` or ad-hoc `rgba(0,60,136,…)` — those were the old values this token replaces.

**Spacing** — use the 4-point scale tokens (`--space-1` 4px … `--space-5` 24px) for all gaps, margins between controls, and paddings. Do not introduce new raw pixel values for spacing.

**Borders** — `var(--color-border)` (`#CCC`) for outer panel/input borders; `var(--color-divider)` (`#E8E8E8`) for internal list separators and track backgrounds. Never use raw `#ddd` or `#eee`.

**Danger/destructive** — `var(--color-danger)` (`#8B0000`) for destructive button backgrounds; `var(--color-danger-text)` for the X clear button and error text; `var(--color-danger-bg)` for error message backgrounds. Never use `#dc3545`, `#721c24`, or `#f8d7da`.

### External APIs used at runtime

- `https://celestrak.org/NORAD/elements/gp.php?GROUP=amateur&FORMAT=tle` — TLE data
- `https://api.open-elevation.com/api/v1/lookup` — elevation for observer location
- `https://db.satnogs.org/api/transmitters/` — fallback transmitter lookup
